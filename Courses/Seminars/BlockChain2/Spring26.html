    Lecture 1 — What Is a Blockchain?

    Topics

    • What is a blockchain for?
      Coordination among many parties when no single trusted entity exists. If a trusted party is available, you don't need a blockchain. Motivate through financial systems, where trust is often absent.
    • The key innovations
      Bitcoin (2009): a public append-only ledger secured by replication and incentives, plus a fixed-supply digital asset. Ethereum (2015): a fully programmable blockchain computer with composability, where on-chain programs can call each other.
    • The ecosystem today
      Key numbers: assets managed by DeFi protocols, transaction volumes, cost comparisons (e.g., international wire transfer vs. stablecoin transfer). What chains developers are building on.
    • The layered architecture
      Following Boneh's stack: consensus/data availability layer → execution engine (blockchain computer) → applications (DApps, smart contracts) → user-facing tools. This layered view becomes the map for the rest of the seminar.
    • Mapping the layers to the seminar
      Consensus layer → lectures 3–7. Ledger and execution models → lectures 8–9. Smart contracts and applications → lectures 10–11. Privacy → lecture 12. Fees, MEV, and PBS → lecture 13.

    Lecture 2 — State Machine Replication I

    Topics

    • Deterministic state machines
      A service modeled as a function from (state, input) → (state, output). If two replicas start in the same state and process the same inputs in the same order, they reach the same state. This determinism is the foundation everything else rests on.
    • Why replicate?
      A single server is a single point of failure. Replication across independent machines provides fault tolerance and availability: the service survives even when some replicas fail.
    • The SMR approach
      Replicate the state machine across n servers. The core problem reduces to ensuring all non-faulty replicas process the same sequence of requests. This decomposes into two sub-problems: agreement (every replica receives every request) and order (every replica processes them in the same sequence).
    • The role of total order
      A totally ordered log is the central abstraction. Once you have a log that everyone agrees on, execution is deterministic and each replica can independently compute the correct state. The consensus problem is, at its core, the problem of building this shared log.
    • Fault models
      Crash faults: a replica simply stops. Byzantine faults: a replica can behave arbitrarily, including sending contradictory messages. The key result: tolerating Byzantine faults requires n > 3f replicas. We state and give intuition for this bound via the Byzantine Generals story.
    • Concrete example
      Walk through a simple replicated key-value store to make the abstraction tangible: how client requests enter the system, how they get ordered, and how replicas execute and respond.

    Lecture 3 — State Machine Replication II

    Topics

    • Safety vs. liveness
      Define precisely. Safety: no two honest replicas disagree. Liveness: every request is eventually processed. The fundamental tension: you want both, but achieving both simultaneously is constrained.
    • The FLP impossibility
      In a fully asynchronous network, no deterministic protocol can guarantee both safety and liveness with even one crash fault. Walk through the bivalence intuition: the adversary can always keep the system undecided by controlling scheduling.
    • Network models
      Synchrony (known bounded delay Δ), asynchrony (no bound), partial synchrony (bound exists but is unknown, or holds only after some unknown GST). Partial synchrony as the practical sweet spot.
    • The DLS result
      Under partial synchrony, consensus is solvable for f < n/3 Byzantine faults. This is the theoretical foundation for PBFT, Tendermint, HotStuff, and all the BFT protocols students will encounter later.
    • The Sybil problem
      All of the above assumes a known, fixed set of n participants. In an open, permissionless setting, one entity can create arbitrarily many identities, breaking the f < n/3 assumption entirely.
    • The bridge to PoW/PoS
      Sybil resistance requires tying identity to a scarce resource (computation, capital). Preview of how Nakamoto consensus and proof-of-stake address this, setting up lectures 5 and 6.

    Lecture 4 — Proof of Work

    Topics

    • Resource-based Sybil resistance
      Recap the Sybil problem from lecture 4. Proof of Work ties voting power to computational effort: one CPU, one vote. This makes it costly to dominate the system.
    • Hash puzzles and mining
      Finding a nonce such that H(block || nonce) < target. The puzzle is hard to solve, easy to verify. Difficulty adjustment: keeping block time roughly constant as total hash power changes.
    • Leader election via PoW
      Whoever solves the puzzle first proposes the next block and earns the block reward. This is a randomized, memoryless lottery — probability of winning is proportional to hash power.
    • The longest-chain rule
      When a node sees multiple valid chains, it follows the longest (most work). This is the fork-choice rule that drives convergence. Honest miners always extend the longest chain they know of.
    • Forks and confirmations
      Forks happen naturally due to network delay (two miners solve nearly simultaneously). They resolve as one chain grows longer. Confirmation depth: waiting k blocks reduces the probability of a reversal exponentially. The 6-confirmation convention.
    • Security intuition
      The honest majority assumption: as long as honest miners control >50% of hash power, the honest chain grows faster in expectation. Brief sketch of why an attacker with <50% falls behind (random walk / gambler's ruin argument).
    • Selfish mining
      A strategy where a miner withholds blocks to gain an advantage. Shows that the true security threshold is slightly below 50%. Briefly discuss the impact on chain quality.

    Lecture 5 — Proof of Stake

    Topics

    • From computation to capital
      PoW ties security to energy expenditure; PoS ties it to economic stake. Validators lock up capital instead of burning electricity. Sybil resistance through scarcity of coins rather than scarcity of hash power.
    • Validator selection and committees
      How validators are chosen to propose and attest to blocks. Randomized selection weighted by stake. The role of committees in scaling attestation (not every validator votes on every slot).
    • BFT-style finality
      Unlike PoW's probabilistic finality, PoS protocols can achieve economic finality: once a block is finalized, reverting it requires destroying a provably large amount of stake. Connection to the BFT protocols from lecture 4 (PBFT, Tendermint, HotStuff).
    • Slashing
      The enforcement mechanism: validators who sign contradictory messages (equivocate) or violate protocol rules have their stake destroyed. This creates a direct economic cost for misbehavior, replacing PoW's energy cost with capital risk.
    • Nothing-at-stake and long-range attacks
      The core theoretical concern: unlike PoW, creating an alternative history costs nothing in old keys. Long-range attacks: an adversary who held stake in the past can try to rewrite history from that point. Why this is fundamentally different from PoW's security model.
    • Weak subjectivity
      The solution to long-range attacks: new or returning nodes must obtain a recent trusted checkpoint (not just the genesis block). This is a real trust assumption that PoW does not require. Tradeoffs and practical implications.
    • Ethereum's Gasper as a case study
      Briefly walk through Ethereum's PoS design: the beacon chain, epochs/slots, LMD-GHOST fork choice combined with Casper FFG finality. Concrete illustration of how the above concepts fit together in a live system.

    Lecture 6 — Federated Byzantine Agreement

    Topics

    • Motivation
      Classical BFT assumes a fixed, known validator set chosen by some authority. Permissionless consensus (PoW/PoS) opens membership but introduces Sybil problems and resource costs. FBA offers a third path: each node individually declares whom it trusts, and global agreement emerges from the overlap of those local trust decisions. No central authority picks the validator set; no resource expenditure is required.
    • Quorum slices and quorums
      Each node defines a quorum slice: a set of peers it considers sufficient for agreement. A quorum is any set of nodes where every member's slice requirement is satisfied within the set. Key insight: quorums are a global, emergent property even though each node only specifies local preferences.
    • Quorum intersection and safety
      Two quorums that share at least one honest node cannot finalize contradictory statements. If quorum intersection holds system-wide, the network enjoys the same safety guarantee as classical BFT — no two honest nodes will disagree. If intersection fails (the network "bifurcates"), independent partitions can diverge without detecting it.
    • Liveness and blocked sets
      A set of nodes is blocked if it depends on faulty or silent nodes to form any quorum. Blocked nodes cannot make progress even if they are honest. FBA therefore does not guarantee liveness for all participants — only for those whose trust graph remains well-connected after removing faulty nodes. Contrast with PoW/PoS, where every honest node with network access eventually makes progress.
    • Stellar Consensus Protocol (SCP)
      Concrete instantiation of FBA. SCP uses federated voting (vote → accept → confirm) and a nomination phase for value proposal followed by a ballot phase for commit/abort, loosely inspired by two-phase commit and Paxos. Walk through a simplified round to show how nomination converges and ballots finalize.
    • Trust graph topology in practice
      Stellar's real network is dominated by a small tier-1 set of organizations whose slices heavily overlap, producing strong quorum intersection but also concentration. Discussion: does FBA in practice converge toward a de-facto permissioned core? Comparison with how PoS stake distribution also tends to concentrate.
    • Tradeoffs summary
      FBA avoids energy cost and explicit stake lockup but introduces a subjective trust assumption. Safety depends on the trust graph staying well-connected; there is no automatic protocol-level punishment for misbehavior (no slashing analogue). Best suited for networks where participants have real-world reasons to identify and trust specific counterparts (e.g., financial institutions, payment anchors).

    Lecture 7 — UTXO

    Topics

    • From consensus to execution
      Lectures 3–7 established how nodes agree on an ordered log of transactions. This lecture asks: what does a transaction actually look like, and what state does the ledger maintain? The UTXO model is Bitcoin's answer: the ledger is a set of unspent outputs, and a transaction consumes some outputs and creates new ones.
    • Anatomy of a Bitcoin transaction
      Inputs reference previous outputs by (txid, index) and provide an unlocking script (scriptSig). Outputs specify an amount and a locking script (scriptPubKey). A transaction is valid if every input correctly unlocks its referenced output, no output is already spent, and the sum of input values ≥ sum of output values (the difference is the fee).
    • The UTXO set as global state
      At any point the ledger state is the set of all unspent transaction outputs. Applying a block means removing the consumed UTXOs and adding the newly created ones. There is no "balance" field — a wallet's balance is simply the sum of UTXOs it can unlock. Implications for how nodes store and index state.
    • Script: Bitcoin's minimal scripting language
      Bitcoin transactions are validated by a simple stack-based, deliberately non-Turing-complete scripting language. Common patterns: Pay-to-Public-Key-Hash (P2PKH), Pay-to-Script-Hash (P2SH), multisig. The language is intentionally limited to keep validation fast and predictable — a design philosophy that contrasts sharply with Ethereum (next lecture).
    • Transaction graph and privacy
      Because every input explicitly references a previous output, the full transaction graph is visible on-chain. This makes Bitcoin pseudonymous, not anonymous: addresses are unlinkable in theory, but heuristics (common-input-ownership, change-output detection) can cluster them. Brief preview of the privacy discussion in Lecture 12.
    • Concurrency-friendliness of UTXOs
      Since each UTXO can only be consumed once and transactions reference specific outputs, two transactions that touch disjoint UTXO sets are trivially independent. This makes parallel validation straightforward and conflict detection simple — a property the account model does not naturally share.
    • Limitations and extensions
      The UTXO model makes stateful computation awkward: there is no persistent storage, no loops, and expressing complex conditions requires chaining multiple transactions. Extensions like Bitcoin's Taproot (Schnorr signatures, MAST) expand expressiveness while preserving the UTXO philosophy. Comparison: why Ethereum chose a fundamentally different model (Lecture 9).

    Lecture 8 — Account Model

    Topics

    • From UTXOs to accounts
      The UTXO model treats the ledger as a set of discrete, one-time-use coins. The account model (introduced by Ethereum) instead maintains a global state mapping addresses to persistent account objects. Each account has a balance, a nonce (to prevent replay), and — for contract accounts — storage and code. Transactions mutate this global state directly.
    • Two kinds of accounts
      Externally Owned Accounts (EOAs) are controlled by a private key and can initiate transactions. Contract accounts hold EVM bytecode and are activated when called. Only EOAs can originate a transaction; contracts execute in response. Implications for the user experience and for composability.
    • State transitions as computation
      A transaction in the account model is not just a transfer; it is a function call that reads and writes global state. The Ethereum Virtual Machine (EVM) is a stack-based, quasi-Turing-complete execution environment. Every operation has a defined gas cost, and a transaction specifies a gas limit. If execution exceeds the limit, all state changes revert but the fee is still paid. This is fundamentally more expressive than Bitcoin Script.
    • Gas as metering
      Gas serves two purposes: (1) it guarantees termination by bounding computation, and (2) it prices the use of shared resources (CPU, storage, bandwidth). Different opcodes cost different amounts of gas. Storage writes (SSTORE) are particularly expensive because they impose a permanent burden on all future full nodes. Brief discussion of gas pricing challenges and historical repricing events (e.g., the Shanghai DoS attacks of 2016).
    • Global state and its costs
      The account model's global state (a Merkle-Patricia trie mapping every account and storage slot) must be maintained by every full node. State growth is a long-term scalability concern: unlike transaction history, state cannot be pruned without losing the ability to validate new blocks. Contrast with UTXO, where the active set is typically smaller and old UTXOs naturally get consumed.
    • Concurrency challenges
      Because transactions can read and write overlapping parts of the global state, determining which transactions are independent is harder than in the UTXO model. Naive sequential execution is the default; parallel execution requires dependency analysis (e.g., Solana's explicit account-access declarations, or Aptos's Block-STM optimistic concurrency).
    • Design comparison: UTXO vs. Account
      UTXO offers natural parallelism, simple verification, and better base-layer privacy (no address reuse by default). The account model offers richer programmability, simpler mental model for developers, and native support for persistent state. Neither is strictly dominant; the choice reflects different priorities (Bitcoin as sound money vs. Ethereum as a programmable platform). This sets up the smart-contracts discussion in Lecture 10.

    Lecture 9 — Smart Contracts

    Topics

    • What is a smart contract?
      A smart contract is a program stored on-chain whose execution is enforced by the consensus protocol. Once deployed, its code is immutable (unless explicitly designed to be upgradeable) and anyone can invoke it by sending a transaction. The blockchain acts as both the execution environment and the court of last resort — there is no external enforcement mechanism. Contrast with traditional legal contracts: code replaces ambiguity, but also replaces flexibility.
    • Programmable transactions and composability
      Smart contracts can call other contracts within a single transaction, enabling arbitrary composition of on-chain logic. A user can atomically swap tokens, deposit collateral, and borrow assets in one transaction by chaining contract calls. This composability is the foundation of the DeFi ecosystem (Lecture 11) and is sometimes called "money legos." It is also a key source of complexity and risk.
    • Development languages and compilation
      Solidity (Ethereum) as the dominant high-level language: syntax, types, storage vs. memory, function visibility. Brief mention of alternatives: Vyper (Python-like, deliberately less expressive), Move (resource-oriented, linear types), Rust-based contracts (Solana, NEAR). All compile down to bytecode executed by a virtual machine. The language shapes which bugs are easy or hard to write.
    • Contract interaction patterns
      Common patterns that recur across DeFi and beyond: token standards (ERC-20 as a minimal fungible-token interface, ERC-721 for NFTs), the approve/transferFrom pattern for delegated spending, and proxy patterns for upgradeability. Standards enable permissionless interoperability: any contract that speaks ERC-20 can integrate with any DEX or lending protocol without coordination.
    • Security pitfall: reentrancy
      Detailed walkthrough of the reentrancy vulnerability. When contract A calls contract B, B can call back into A before A's state updates are complete, violating A's invariants. The 2016 DAO hack as the canonical example: an attacker drained roughly 3.6 million ETH by repeatedly re-entering a withdrawal function. Defenses: the checks-effects-interactions pattern, reentrancy guards (mutexes), and pull-over-push withdrawal designs. Why this class of bug persists despite being well-known.
    • Broader vulnerability landscape
      Brief survey of other common pitfalls: integer overflow/underflow (largely mitigated since Solidity 0.8), oracle manipulation (feeding a contract a false price), access-control errors (unprotected admin functions), and front-running (transaction ordering exploits, previewing Lecture 13). The common thread: smart contracts operate in an adversarial, permissionless environment where every external input and every ordering assumption can be exploited.
    • Auditing, verification, and the limits of "code is law"
      The practice of security audits, bug bounties, and formal verification tools (e.g., Certora, Halmos). Despite these, exploits remain frequent because audit coverage is incomplete and economic incentives to attack are high. Discussion of the tension between immutability ("code is law") and the practical need for upgradeability, governance, and emergency pauses. The DAO fork as a case study in social consensus overriding on-chain finality.

    Lecture 10 — DeFi Applications

    Topics

    • What is DeFi?
      Decentralized finance replaces trusted intermediaries (banks, brokers, exchanges) with smart contracts that execute financial logic on-chain. Anyone can interact with these contracts permissionlessly, and the rules are publicly auditable. Key promise: composability turns individual primitives into a programmable financial stack. Key risk: bugs, oracle failures, and economic exploits propagate through the same composability.
    • Automated Market Makers (AMMs)
      Traditional exchanges use order books; AMMs replace them with a deterministic pricing function over liquidity pools. Constant-product markets (Uniswap v2): two-token pools where x · y = k determines the price. Liquidity providers deposit both assets and earn fees; in return they bear impermanent loss when prices move. Uniswap v3's concentrated liquidity: LPs choose a price range, increasing capital efficiency but also complexity and active management.
    • Lending and liquidation
      Overcollateralized lending protocols (Aave, Compound): users deposit collateral and borrow up to a loan-to-value ratio. Interest rates are set algorithmically as a function of pool utilization. When a borrower's collateral value falls below the liquidation threshold, any third party can repay part of the debt and seize collateral at a discount. Liquidations keep the system solvent but can cascade during sharp market drops, creating systemic stress.
    • Stablecoins
      Tokens designed to track a reference price (usually $1). Three families: (1) fiat-backed / custodial (USDC, USDT) — simple but centralized and censorable; (2) crypto-collateralized (DAI / MakerDAO) — overcollateralized vaults, governance-set parameters, liquidation mechanisms; (3) algorithmic / undercollateralized — attempt to maintain the peg via elastic supply or arbitrage incentives, historically fragile (Terra/UST collapse as case study). Stablecoins are the most widely used DeFi primitive and the critical bridge to traditional finance.
    • Oracles
      Many DeFi contracts need off-chain data (asset prices, interest rates). An oracle is any mechanism that brings external information on-chain. Chainlink's decentralized oracle network as the dominant design: multiple independent node operators report data, and an aggregation contract produces a canonical answer. Oracle manipulation attacks: if an attacker can move the reported price — by manipulating a low-liquidity spot market or exploiting a stale feed — they can trigger profitable liquidations or drain lending pools. The oracle problem is fundamental: blockchains are deterministic and isolated by design.
    • Composability in action — and when it breaks
      A concrete example of DeFi composition: flash loans. A user borrows an arbitrary amount with no collateral, executes a sequence of contract calls, and repays within the same transaction — if repayment fails, the entire transaction reverts atomically. Flash loans enable capital-free arbitrage and liquidation, but they also amplify exploits by giving any attacker effectively unlimited capital for the duration of one transaction. Walk through a simplified flash-loan-funded oracle-manipulation attack.
    • Systemic risk and open questions
      DeFi protocols share collateral, liquidity, and oracle feeds, creating hidden interdependencies. A failure in one protocol can cascade: the collapse of Terra/UST in May 2022 triggered liquidations across multiple lending markets and destabilized several other stablecoins. Open questions: how to measure systemic risk in a permissionless system, whether governance can respond fast enough to crises, and the growing tension between decentralization and regulatory compliance.

    Lecture 11 — Privacy in Cryptocurrencies

    Topics

    • The privacy problem
      Public blockchains are radically transparent: every transaction, amount, and address is permanently visible to everyone. Bitcoin and Ethereum are pseudonymous, not anonymous — once a pseudonym is linked to a real identity (through an exchange, a payment, or metadata), the entire transaction history is exposed. Blockchain analytics firms (Chainalysis, Elliptic) routinely deanonymize users. Privacy is not just a personal preference; it is a prerequisite for fungibility (if coins with certain histories are worth less, the currency is not truly fungible).
    • Deanonymization techniques
      Common-input-ownership heuristic: inputs spent together likely belong to the same wallet. Change-output detection: identifying which output returns value to the sender. Timing and network-level analysis: correlating transaction broadcast times with IP addresses. These heuristics are imperfect but effective at scale, especially when combined with off-chain data (exchange KYC records, public addresses posted online). Connection back to the UTXO transaction graph discussion in Lecture 8.
    • Mixing and CoinJoin
      The simplest privacy technique: multiple users combine their inputs and outputs into a single transaction, breaking the link between sender and receiver. CoinJoin (Bitcoin): a collaborative protocol where participants jointly construct a transaction with equal-valued outputs. Limitations: requires coordination, equal denominations reduce flexibility, and sophisticated analysis can sometimes unmix. Centralized mixers (tumblers) add counterparty risk. Wasabi Wallet and JoinMarket as practical implementations.
    • Ring signatures and stealth addresses
      Monero's approach: each transaction input is signed using a ring signature that includes the real input and several decoy inputs drawn from the blockchain. An observer cannot determine which ring member is the true spender. Stealth addresses: the sender derives a one-time destination address for every payment, so the recipient's public address never appears on-chain. Together, ring signatures hide the sender and stealth addresses hide the receiver.
    • Confidential Transactions
      Hiding amounts: Pedersen commitments allow a transaction to commit to an output value without revealing it, while a range proof guarantees the committed value is non-negative (preventing inflation). Used in Monero (RingCT combines ring signatures with confidential amounts) and in Elements/Liquid (Bitcoin sidechain). Tradeoff: range proofs add significant size to transactions — Bulletproofs reduced this cost substantially but did not eliminate it.
    • Zero-knowledge proofs for privacy
      Zcash's approach: a shielded transaction proves in zero knowledge that the sender owns unspent notes, that the transaction balances, and that no double-spend occurs — without revealing sender, receiver, or amount. The blockchain stores only nullifiers (spent-note identifiers) and commitments. Originally used zk-SNARKs with a trusted setup (Groth16); newer proposals use transparent setups (Halo 2). Maximum theoretical privacy, but the shielded pool's anonymity set is only as large as the number of users who actually opt in — and in practice most Zcash transactions remained transparent.
    • Privacy Pools
      A proposal (Buterin, Illum, Nadler, Schär, Soleimani, 2023) to reconcile privacy with regulatory compliance. Users transact through a mixing pool — preserving unlinkability — but attach a zero-knowledge proof of membership in an association set, demonstrating that their funds do not originate from sanctioned or illicit sources without revealing the specific transaction history. This separates the privacy guarantee (hiding the link between deposit and withdrawal) from the compliance guarantee (proving legitimacy of origin). Association sets can be curated by different providers with different inclusion criteria, allowing a spectrum of compliance levels. Open questions: who defines the association sets, what happens when sets disagree, and whether regulators will accept probabilistic compliance.
    • Tradeoffs summary
      Every privacy technique carries costs: larger transactions (ring signatures, range proofs), latency (mixing coordination), smaller anonymity sets (shielded pools that few use), reduced auditability, or additional trust assumptions (trusted setups). There is no free privacy — each design point trades off cost, usability, and strength of the guarantee. The fundamental tension remains: blockchains are designed for transparency and verifiability, while privacy requires hiding information. How much of this tension can be resolved by cryptography (Privacy Pools, selective disclosure) versus how much is irreducibly a policy and social question is an open debate.

    Lecture 12 — Fees, MEV & PBS

    Topics

    • Blockspace as a scarce resource
      Every block has a capacity limit (gas limit in Ethereum, weight in Bitcoin). When demand exceeds supply, the protocol needs a mechanism to allocate space. Transaction fees serve this role: users bid for inclusion, and the protocol (or the block producer) decides which transactions to include. This makes blockspace a market, and fee mechanism design is market design.
    • First-price auctions and their problems
      Bitcoin and pre-EIP-1559 Ethereum used a simple first-price auction: users set a gas price, and miners included the highest-paying transactions. Problems: users have no good strategy for choosing a bid (overpay or risk not being included), fees are volatile and hard to predict, and the mechanism is not incentive-compatible — rational users waste effort estimating the minimum winning bid.
    • EIP-1559 and dynamic base fees
      Ethereum's fee reform (August 2021): each block has a protocol-computed base fee that adjusts up or down depending on whether the previous block was above or below a target size. The base fee is burned (not paid to the proposer). Users additionally include a priority fee (tip) to incentivize the proposer. Key properties: fee estimation becomes trivial in normal conditions (just pay the base fee plus a small tip), the mechanism is approximately incentive-compatible, and burning the base fee aligns monetary policy with network usage. Connection to mechanism design: EIP-1559 can be understood as a posted-price mechanism with a dynamic reserve.
    • Maximal Extractable Value (MEV)
      The block producer chooses not just which transactions to include but in what order. This ordering power is economically valuable. MEV is the total value a block producer can extract by optimally ordering, inserting, or censoring transactions. Examples: DEX arbitrage (inserting a trade that corrects a price discrepancy), sandwich attacks (front-running and back-running a large swap to profit from the price impact), and liquidation racing (being first to liquidate an undercollateralized position). MEV exists because DeFi applications leak value through publicly visible pending transactions.
    • The mempool as an information channel
      Pending transactions sit in a public mempool before inclusion. Searchers — sophisticated actors who monitor the mempool — detect MEV opportunities and compete to capture them by submitting their own transactions with precise ordering requirements. This competition can be wasteful (priority gas auctions that bid up fees and congest the network) and harmful to users (sandwich attacks extract value directly from ordinary swaps). The mempool turns blockchains into an adversarial information environment.
    • Proposer–Builder Separation (PBS)
      A structural response to MEV concentration. In naive PoS, the validator who proposes a block also builds it, creating an incentive to run sophisticated MEV-extraction strategies — which favors large, well-resourced operators and threatens decentralization. PBS separates the roles: builders compete to construct the most valuable block, and the proposer simply selects the highest-paying block via a sealed-bid auction (MEV-Boost in Ethereum today). The proposer need not know what is inside the block. This preserves validator decentralization but concentrates building power — the builder market is currently dominated by a small number of firms.
    • Open problems and current research
      Builder centralization: a small number of builders win most blocks, raising censorship concerns. Inclusion lists (FOCIL): a proposal to let proposers force-include certain transactions, limiting builders' censorship power. Encrypted mempools and order-fair sequencing: hiding transaction content until ordering is committed, reducing front-running. The broader question: can mechanism design tame MEV, or is some extraction an unavoidable cost of permissionless, composable finance?
